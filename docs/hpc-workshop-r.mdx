### 1.2 What is High-Performance Computing (HPC)?

HPC refers to powerful computing systems designed to solve complex computational problems that require massive processing power — far beyond what a typical desktop or laptop can provide.

![What is HPC?](/img/workshop-r/what-is-hpc.png)

#### HPC in Action: Medical Research
- **Accelerating Drug Discovery** — Simulating complex molecular interactions (e.g., AlphaFold2) to drastically reduce drug development timelines.
- **AI-Driven Healthcare** — Powering AI models that analyze medical imaging (e.g., X-ray heatmaps) for faster, more accurate diagnosis.
- **Large-Scale Omics Analysis** — Processing massive biological datasets such as genomes, proteomes, and metabolomes.

#### Understanding an HPC Cluster

![HPC Cluster](/img/workshop-r/hpc-cluster.png)

An **HPC cluster** is made up of many individual computers called **nodes**. Each node has its own CPU, memory, local storage, and network interface. All nodes share fast storage volumes (`/scratch` and `/project`) and are managed by a job scheduler called **SLURM** from the **login node**.

### 1.3 MedCMU-HPC Cluster Overview

The **Raptor** cluster empowers medical research with scalable computing for **Omics** and **AI** workloads.

![RAPTOR cluster overview](/img/workshop-r/raptor-cluster-overview.png)

#### Cluster Partitions

The cluster is organized into **partitions** — groups of nodes reserved for specific purposes:

| Partition   | Total Cores | Cores per Node | Memory per Node | GPU per Node    | Max Time per Job | Max Cores per User |
|-------------|-------------|----------------|-----------------|-----------------|------------------|--------------------|
| **short**   |             | 1              | 8 GB            | -               | 8 hours          | 2                  |
| **compute** | 384         | 94             | 1,280 GB        | -               | -                | 188                |
| **gpu**     | 188         | 94             | 384 GB          | 2 × Nvidia H100 | -                | 188                |

:::tip When to use each partition
- **short** — Quick tests, interactive sessions, and array jobs (default: 4 GB/core).
- **compute** — Large multi-node or long-running jobs.
- **gpu** — GPU-accelerated workloads (deep learning, molecular simulation).
:::

### 3.6 Tracking Remaining Project Resources

Use these built-in commands to monitor your remaining compute time and storage:

**Check remaining compute hours:**
```bash

```
```text title="Example output"
Account    Descr   Remaining(%)  Allocation(SHr)        Used(SHr)   Remaining(SHr)
----------------------------------------------------------------------------------
a25000X    ProjA       100.00%           100.00             0.00           100.00
```

Service Hours (SHr) are calculated as:

$$
SHr = \frac{Cores \times Hours}{Total\ Cores\ per\ Node}
$$

| Example | CPU (cores) | Running Time (h) | SHr Used |
|---------|-------------|-------------------|----------|
| User B1 | 1           | 2                 | $\frac{1 \times 2}{94} = 0.02$ |
| User C3 | 94          | 1                 | $\frac{94 \times 1}{94} = 1.00$ |

## 4. Submitting Jobs with SLURM (Basic)

> On an HPC cluster, you don't run programs directly — you **submit jobs** to the SLURM scheduler, which queues and allocates resources for you. This section covers interactive jobs and basic batch scripts.

### 4.1 Interactive Jobs

Interactive jobs give you a shell on a compute node in real time. This is useful for testing and debugging.

```bash
# Basic interactive session (4 GB memory)
srun --mem=4G --pty bash
```
```bash
# With specific partition, time limit, cores, and memory
srun -p short -t 1:00:00 -c 1 --mem=4G --pty bash
```

**Run an R script interactively on a compute node:**
```bash
srun -p short -t 1:00:00 -c 1 --mem=4G --pty \
   Rscript /common/demo/mean_score.r 1 2 3 4 5
```

:::info Time limit format
Wall time is specified as `d-hh:mm:ss` — for example, `1:00:00` = 1 hour, `1-00:00:00` = 1 day.
:::

### 4.2 How SLURM Batch Jobs Work

For longer or unattended jobs, you write a **batch script** and submit it with `sbatch`. SLURM handles the rest automatically.

![SLURM submit process](/img/workshop/slurm-submit.png)

1. **Log in** to the cluster via SSH
2. **Write a script** specifying cores, memory, time, and your program
3. **Submit** the script — SLURM adds it to the job queue
4. **SLURM allocates** resources when available and runs your job
5. **Output is saved** to log files when the job completes

### 4.3 Your First SLURM Batch Script

Create a file called `1_MeanScore_script.sh`:

```bash title="1_MeanScore_script.sh"
#!/bin/bash

#SBATCH --job-name=MeanScore
#SBATCH --account=w260001
#SBATCH --partition=short
#SBATCH --time=1:00:00
#SBATCH --cpus-per-task=1
#SBATCH --mem=4G
#SBATCH --output=1_MeanScore_%j.out
#SBATCH --error=1_MeanScore_%j.err

module load R
Rscript /common/demo/mean_score.r 1 2 3 4 5
```

Submit it:
```bash
sbatch 1_MeanScore_script.sh
```

:::tip
Use the [interactive SLURM Script Generator](Tools/slurm-generator) to create batch scripts without memorizing the syntax.
:::

## 6. Advanced SLURM: Array Jobs

> When you need to run the **same script with different inputs** (e.g., processing multiple datasets), **array jobs** let you submit them all at once. SLURM automatically manages parallelism and resource allocation.

![SLURM Array Job](/img/workshop-r/array-job.png)

### 6.1 How Array Jobs Work

An array job creates multiple sub-tasks from a single script. Each sub-task receives a unique index via `$SLURM_ARRAY_TASK_ID`, which you use to select different inputs.

### 6.2 Example: Array Job with R

```bash title="3_MeanScore_script_array.sh"
#!/bin/bash

#SBATCH --job-name=MeanScore_Array
#SBATCH --account=w260001
#SBATCH --partition=short
#SBATCH --time=1:00:00
#SBATCH --cpus-per-task=1
#SBATCH --mem=8G
#SBATCH --output=3_MeanScore_array_%A_%a.out
#SBATCH --error=3_MeanScore_array_%A_%a.err
#SBATCH --array=0-2

# Define different inputs for each array task
datasets=("10 20 30" "40 50 60" "70 80 90")

INPUT_ARGS=${datasets[$SLURM_ARRAY_TASK_ID]}

module load apptainer r

echo "Task $SLURM_ARRAY_TASK_ID: Processing $INPUT_ARGS on $(hostname)"
apptainer exec $SIF Rscript /common/demo/mean_score.r $INPUT_ARGS
```

Submit it:
```bash
sbatch 3_MeanScore_script_array.sh
```

This creates **3 sub-tasks** (indices 0, 1, 2), each computing the mean of a different set of numbers. Output files are named with both the job ID (`%A`) and the array index (`%a`) for easy identification.

:::tip Array job notation
- `--array=0-9` — 10 tasks (indices 0 through 9)
- `--array=0-9%2` — 10 tasks, but only 2 run at a time
- `--array=1,3,5,7` — 4 specific tasks
:::

## 7. Hands-on Practice: Precision Health Planning

> **Scenario**: You are part of the national Precision Health steering committee. To design health interventions tailored for different provinces and generations (e.g., Baby Boomers vs. Gen Z), you need to analyze the obesity (BMI) distribution of over 700,000 citizens across 6 provinces.
> The datasets are huge (over 100,000 records per province). You will use **SLURM Array Jobs** and the **tidyverse** package (inside an R container) to process them in parallel, then synthesize the results in an R Markdown document.

### 7.1 Prepare the Workspace
First, create a directory for this practice session and copy the provincial health records.

```bash
mkdir -p ~/bmi_practice
cd ~/bmi_practice

# Copy the practice datasets and R script
cp -r /common/demo/clinical_data/* .

# Verify the files are copied
ls -l
```
*(You should see 6 large province datasets: `cohort_1.csv` to `cohort_6.csv`, and an R script `process_bmi.R`)*

### 7.2 Understand the R Script
The script uses `tidyverse` to parse dates, calculate BMI, classify the generation, and produce a summarized report. Let's look at the core mapping logic:

```bash
cat process_bmi.R
```

```r title="process_bmi.R snippet"
# process_bmi.R snippet
args <- commandArgs(trailingOnly = TRUE)
input_file <- args[1]
output_file <- args[2]

library(tidyverse)
data <- read_csv(input_file, show_col_types = FALSE)

# Calculate Age, classify Generation, and calculate Asian BMI cutoff
data_processed <- data %>%
  mutate(
    BirthYear = as.numeric(substr(BirthDate, 1, 4)),
    CaptureYear = as.numeric(substr(CaptureDate, 1, 4)),
    Age = CaptureYear - BirthYear,
    Generation = case_when(
      BirthYear >= 1997 ~ "Gen Z",
      BirthYear >= 1981 ~ "Gen Y",
      BirthYear >= 1965 ~ "Gen X",
      BirthYear >= 1946 ~ "Baby Boomer",
      TRUE ~ "Silent Gen"
    ),
    BMI = WeightKg / (HeightCm/100)^2,
    BMI_Category = case_when(
      BMI < 18.5 ~ "Underweight",
      BMI < 23.0 ~ "Normal",
      BMI < 25.0 ~ "Overweight",
      BMI < 30.0 ~ "Obese I",
      TRUE ~ "Obese II"
    )
  )

# Output only the summary statistics to save time and disk space!
summary_stats <- data_processed %>%
  group_by(Province, Generation, BMI_Category) %>%
  summarise(Total = n(), Avg_Age = mean(Age), Avg_BMI = mean(BMI), .groups = "drop")

write_csv(summary_stats, output_file)
```

### 7.3 Write the SLURM Array Job
Create a batch script named `submit_bmi_analysis.sh` to run this R script on all 6 province files at once.

```bash title="submit_bmi_analysis.sh"
#!/bin/bash
#SBATCH --job-name=PrecisionBMI
#SBATCH --account=w260001
#SBATCH --partition=short
#SBATCH --time=00:30:00
#SBATCH --cpus-per-task=1
#SBATCH --mem=8G
#SBATCH --output=logs/bmi_job_%A_%a.out
#SBATCH --error=logs/bmi_job_%A_%a.err
#SBATCH --array=1-6

# Create logs directory
mkdir -p logs

# Dynamic input/output for each task ID
INPUT_CSV="cohort_${SLURM_ARRAY_TASK_ID}.csv"
SUMMARY_CSV="summary_province_${SLURM_ARRAY_TASK_ID}.csv"

echo "Processing $INPUT_CSV on node $(hostname)"

# Load modules
module load apptainer r

# Run the analyzer inside the reproducible container
apptainer exec $SIF Rscript process_bmi.R $INPUT_CSV $SUMMARY_CSV

echo "Successfully wrote summary to $SUMMARY_CSV"
```

### 7.4 Submit and Monitor
Submit the array job to the SLURM scheduler:
```bash
sbatch submit_bmi_analysis.sh
```

Monitor your job queue:
```bash
watch squeue -u $USER
```

Once the jobs complete, check the results. You will see 6 new `summary_province_*.csv` files outlining the statistical spread!

### 7.5 Compile Strategies in RStudio (R Markdown)

Now that the heavy lifting is done by the HPC compute nodes locally mapping the huge data, we can switch back to the web portal to aggregate the visual reports!

1. Go back to OnDemand.
2. Launch an **RStudio interactive session**.
3. In RStudio, open a **New R Markdown (`.Rmd`) Document**.
4. In the R Markdown file, execute the following script to aggregate the SLURM outputs and formulate behavioral strategies!

```r title="analyze_strategies.Rmd"
library(tidyverse)
library(ggplot2)

# 1. Read all 6 summarized files into one combined dataset
files <- list.files(path = "~/bmi_practice", pattern = "summary_province_.*\\.csv", full.names = TRUE)
combined_stats <- map_dfr(files, read_csv, show_col_types = FALSE)

# 2. Descriptive Statistics: Overall count by Province and Generation
overview <- combined_stats %>%
  group_by(Province, Generation) %>%
  summarise(Total_Citizens = sum(Total),
            Avg_BMI = weighted.mean(Avg_BMI, Total), 
            .groups = "drop") %>%
  arrange(desc(Total_Citizens))

cat("\n--- Descriptive Statistics: Population Overview ---\n")
print(head(overview, 10))

# 3. Data Visualization: Obesity (Obese I & II) Distribution by Generation
obesity_data <- combined_stats %>%
  filter(BMI_Category %in% c("Obese I", "Obese II")) %>%
  group_by(Province, Generation) %>%
  summarise(Obese_Count = sum(Total), .groups = "drop")

ggplot(obesity_data, aes(x = Generation, y = Obese_Count, fill = Province)) +
  geom_bar(stat = "identity", position = "dodge") +
  theme_minimal() +
  labs(title = "Obesity Distribution by Generation and Province",
       subtitle = "High-Risk Target Groups for Precision Interventions",
       x = "Generation",
       y = "Total Obese Citizens (Obese I & II)") +
  theme(axis.text.x = element_text(angle = 45, hjust = 1))

# 4. Precision Strategic Health Plans
cat("\n========== STRATEGIC HEALTH PLANS ==========\n")
cat("1. Baby Boomers (~Chiang Mai): Implement low-impact community exercise (e.g., Tai Chi) and diet clinics.\n")
cat("2. Gen X (Widespread): Introduce workplace 'Active Office' campaigns and mandatory annual cholesterol/BMI screenings.\n")
cat("3. Gen Z (~Bangkok): Focus on limiting screen time, mental health pairing, and app-based gamified fitness challenges.\n")
```
*(You've successfully handled massive raw data using HPC compute nodes, generated descriptive stats & figures, and synthesized the results using RStudio Web!)*
